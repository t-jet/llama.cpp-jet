## Build defect detail

The QA pre-flight relink of build-cov against the merged sources failed. The
defect is a git 3-way merge semantic conflict that git did not flag because
the changes are in different files (struct declaration in .h vs. function
body in .cpp).

Error log: ._design_docs/.test_reports/build-cov-relink-stage14-step5-20260611-01.log

Primary error: server-context.cpp(4160,32): error C2440 function-style-cast:
cannot convert from 'initializer list' to 'server_context_meta'. Cascade
errors at lines 4174, 4175, 4177, 4179, 4186, 4188, 4189. Earlier reported
diagnostic: server-context.cpp(1649,13): error C2679: binary '='.

Cause: upstream parent (18ef86ece) added `bool has_inp_video;` to the struct
`server_context_meta` in tools/server/server-context.h (line 29). The local
parent (02db7a768) did not add the matching initializer in
`server_context::get_meta()` return list in tools/server/server-context.cpp.
Git's 3-way merge took upstream's struct change verbatim. The init list at
lines 4160-4191 of server-context.cpp initializes 10 fields of
`server_context_meta` (has_mtmd, has_inp_image, has_inp_audio, json_ui_settings,
...) and is missing has_inp_video, so the brace-enclosed init list maps the
wrong values to the wrong fields and the compiler emits a type-mismatch
cascade.

Upstream commit that added has_inp_video: 8f83d6c27 (mtmd : add video input
support, #24269, in the 94-commit range). Local parent did not anticipate
the new struct field.

## Why this blocks T114 and T114a

- The 9 focused test binaries in build-cov/bin/Release are dated 2026-06-07
  and are linked against the pre-merge static .lib files. The .lib files
  themselves are dated 2026-06-11 (incremental rebuild of llama.lib on
  2026-06-11 15:52), so the test executables execute pre-merge code while
  the static libraries are post-merge.
- A relink (cmake --build build-cov) triggers recompilation of
  server-context.cpp against the merged sources, and the compile fails.
  Without a relink, the test binaries still link against the old static
  .lib and would execute pre-merge code paths; the coverage run would not
  measure the merged tree.
- The Step 3 ctest 61/69 (88%) cited in the cycle state was run on the
  build/ tree (BUILD_SHARED_LIBS=ON), where the .obj files are dated
  2026-06-10 15:17 (pre-merge). build/ was not rebuilt on 2026-06-11. The
  Step 3 numbers are pre-merge measurements, not on the merged tree.

## Step 5 line 571 follow-up (2026-06-11)

### Verdict

PRE-EXISTING_TEST_BUG (Architect). 1-line test fix applied. build

### Fix applied

Commit 998ae00fa. Diff (`tests/test-cache-controller.cpp:560-585`):

```text
@@ tests/test-cache-controller.cpp line 565 (test_hybrid_rejects_partial_blob_match):
-    ctrl.debug_add_entry_for_tests(create_tokens({1, 2, 3, 4}));
-
+    // Stage 14 line 571 fix: use metadata form so entry namespace matches lookup.
     prepared_prompt_metadata meta;
-    GGML_UNUSED(meta);
+    ctrl.debug_add_entry_for_tests(create_tokens({1, 2, 3, 4}), meta);
+
```

The Architect's recommended diff used the 5-arg form
`(tokens, false, "", 64, 0)`. Verification showed that 5-arg
form with `""` namespace falls back to `compute_namespace_id()`
(no metadata), which produces a different namespace string
from what `debug_find_match_tokens_for_tests` computes via
`compute_namespace_id(metadata)`. The strict namespace
compatibility check in `find_best_match` would skip the
entry even when admitted. The metadata form
(`debug_add_entry_for_tests(tokens, meta)`) uses
`compute_namespace_id(metadata)` directly, matching the
lookup namespace, and admits a 1-byte target payload that
passes the Stage 5 "missing target payload" check.
