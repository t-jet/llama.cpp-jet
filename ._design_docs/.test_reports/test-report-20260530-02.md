## Failure analysis: test-step8-integration-wiring, test 10

### Test name

`test_cold_store_wired_to_io_worker` (test-step8-integration-wiring.cpp, line 328)

### Symptom

The test calls `ctrl.demote_payload(1)` to submit an async demotion, then immediately calls `ctrl.process_completions()` without waiting for the I/O worker thread to finish the write. The assertion `TEST_ASSERT(stats["n_demotion_successes"] == 1)` fails because the async write has not completed yet, so `n_demotion_successes` is 0.

### Reproduction

- Release: FAIL (TEST_ASSERT exits with code 1)
- Debug: FAIL (TEST_ASSERT exits with code 1)

The test fails in both Debug and Release configurations. This is not a Release-only issue.

### Root cause

The test has a race condition: it calls `process_completions()` immediately after `demote_payload()` without giving the I/O worker thread time to complete the async write. Other tests in the same file (and in test-step7-promotion-protocol) use `std::this_thread::sleep_for()` before calling `process_completions()` to allow the worker thread to finish. This test does not include a sleep.

This bug was previously masked because `assert()` was a no-op in Release mode (the `#undef NDEBUG` was placed after `#include <cassert>`). The `TEST_ASSERT()` fix correctly exposed this real test bug.

### Recommended fix

Add a `std::this_thread::sleep_for(std::chrono::milliseconds(200))` call between `demote_payload()` and `process_completions()`, consistent with the pattern used in test-step7-promotion-protocol.cpp and other async tests in test-step8-integration-wiring.cpp.
