## Root cause: semantic conflict in the real merge

The real merge commit `e0f3f868b` has two parents:

- `3b9ed9712` (cache-optimization tip, first parent)
- `6ddc9430b` (`upstream_master` tip, second parent)

`tools/server/server-context.cpp` was reported as having no textual
conflict at the merge and "54 lines added with no conflict" per the
real-merge correction record at
[part-15-real-merge-correction.md](../../cache-handling-phase11-implementation/part-15-real-merge-correction.md).
The merge resolver recorded three textual conflicts:
`common/fit.h`, `common/speculative.cpp`, and
`ggml/src/ggml-backend-meta.cpp`. None of those three files contain
a compile error (verified by re-running their diffs against upstream
and confirming the expected `common_get_device_memory_data`,
`common_get_mtp_ctx_memory_overhead`, and `nextn` masked-flag content
is present in the merge tree).

`server-context.cpp` carried a semantic conflict that git's 3-way
merge did not flag. Both branches added a `const bool
near_prompt_end` declaration in the same loop body:

| Branch | Commit that added the line | Line in that branch's tree |
| --- | --- | --- |
| `upstream_master` (`6ddc9430b`) | `e2ef8fe42` ("server: fix checkpoints creation", upstream PR #22929) | 3058 |
| `cache-optimization` (`3b9ed9712`) | Stage 10 chain | 3636 |

The merge algorithm placed both declarations at adjacent lines
(3651 and 3653) in the merged tree. The C++ compiler rejects the
file as a redefinition.

Evidence: the three resolved-conflict files
(`common/fit.h`, `common/speculative.cpp`,
`ggml/src/ggml-backend-meta.cpp`) match the expected merge content
and contain no compile error.

The fabricated single-parent commit `72cfbcd44` (on which the
2026-06-04 closure was based) is a child of `bdb166ac1` (Stage 10
tip) and never went through the merge machinery. It carries only
the cache-optimization content of `server-context.cpp` and has
exactly one `near_prompt_end` declaration at line 3600. That is why
the prior closure builds clean. The real merge tree does not.
