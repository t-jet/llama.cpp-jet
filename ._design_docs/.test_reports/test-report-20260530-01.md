## Failure analysis: test-step3-4-cold-store-write-read

### Symptom

The test binary crashes with exit code 0xC0000409 (STATUS_STACK_BUFFER_OVERRUN) immediately after printing "step3: write with target_and_draft pair..." in Release and RelWithDebInfo configurations. The Debug configuration passes all 27 tests.

### Reproduction

- Release: crash at test 3.6
- RelWithDebInfo: crash at test 3.6
- Debug: all 27 tests pass

### Root cause hypothesis

The crash occurs at the first test that writes a cold file with non-empty draft bytes (test 3.6: `store.write(300, target, draft, snap)` where `draft = {0x04, 0x05}`). The first five tests use empty draft vectors and pass in all configurations.

The STATUS_STACK_BUFFER_OVERRUN exit code indicates a stack cookie check failure, which is typically caused by a buffer overflow on the stack. Possible causes:

1. The `#undef NDEBUG` directive in the test file is placed after `#include <cassert>`, making all `assert()` calls no-ops in Release mode. This means error-checking assertions are silently disabled, potentially allowing undefined behavior to proceed unchecked.

2. A format string mismatch in the `SRV_DBG` logging macro could cause stack corruption via variadic argument processing. The `SRV_DBG` call in the `write` function uses `%zu` for `size_t` arguments, which may interact incorrectly with the variadic `common_log_add` function on MSVC in optimized builds.

3. An MSVC optimization bug or undefined behavior that is masked by Debug mode's different memory layout and active assertions.

### Evidence

- The temp directory `T:\tmp\cold_store_step34_write_pair` exists after the crash but is empty, suggesting the crash occurs before or during the `write` call.
- The Debug build passes all tests, confirming the code logic is correct.
- The crash is reproducible in both Release and RelWithDebInfo configurations.

### Recommended investigation

1. Move `#undef NDEBUG` before `#include <cassert>` in all test files to ensure assertions are active in Release builds.
2. Add explicit error checking (not relying on `assert`) for critical operations like `configure_store` return values.
3. Investigate the `SRV_DBG` format string for potential `%zu` vs `PRIu64` mismatches on MSVC.
4. Consider running the test under Address Sanitizer or with `/RTC1` runtime checks to identify the exact stack corruption point.
